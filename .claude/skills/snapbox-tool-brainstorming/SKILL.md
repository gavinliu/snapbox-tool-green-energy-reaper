# 将 Snapbox Tool 创意头脑风暴转化为设计

## 概述

你通过自然的协作对话，帮助将想法转化为完整的设计和规范。

首先要了解当前项目的背景，然后逐个提出问题来完善想法，一旦你理解了你要构建的内容，就将设计分成几个小部分（200-300字）进行展示，并在每个部分之后检查其内容是否正确。

## 检查清单

你必须为每一项都创建一个任务，并按顺序完成它们：

1. **探索项目背景** - 查看文件、文档、最近的提交记录
2. **逐一提出澄清问题**，了解目的/限制条件/成功标准
3. **提出 2-3 种方案** - 说明其优缺点以及您的建议
4. **呈现设计方案** - 根据复杂程度分章节呈现，并在每个章节完成后征得用户批准
5. **编写设计文档** - 保存到 `docs/specs/YYYY-MM-DD-<topic>-design.md` 并提交
6. **规范自检** - 快速内联检查占位符、矛盾之处、歧义和范围（见下文）
7. **用户审阅规范文档** - 在用户继续操作之前，请用户审阅规范文件
8. **过渡到实施阶段** - 运用 `snapbox-tool-writing-plans` 写技能制定实施计划

## 流程

**澄清需求：**

- 首先查看当前项目状态（文件、文档、最近的提交）
- 在提出详细问题之前，请先评估项目范围：如果需求描述了多个独立的子系统（例如，”构建一个包含聊天、文件存储、计费和分析功能的平台”），请立即标记出来。不要把时间浪费在完善一个需要先分解的项目细节上
- 尽量选择题，但开放式问题也可以接受
- 每条消息只提一个问题——如果某个主题需要更深入探讨，请将其拆分成多个问题
- 重点在于理解：目的、限制条件、成功标准

**探索方法：**

- 提出2-3种不同的方法，并说明每种方法的优缺点
- 以对话的方式提出选项，并附上您的建议和理由
- 首先提出你推荐的方案，并解释原因

**设计方案展示：**

- 一旦你确信自己理解了正在构建的内容，就可以展示设计方案了
- 根据各部分的复杂程度调整篇幅：如果内容简单明了，几句话即可；如果内容微妙复杂，则最多可使用 200-300 字
- 涵盖内容：架构、组件、数据流、错误处理、测试

**设计旨在隔离和清晰：**

- 将系统拆分成更小的单元，每个单元都有明确的目标，通过定义良好的接口进行通信，并且可以独立地被理解和测试。
- 对于每个单元，你应该能够回答：它的功能是什么，如何使用它，以及它依赖于什么？
- 不看内部代码，有人能理解一个单元的功能吗？能在不影响其他用户的情况下修改内部代码吗？如果不能，那么边界条件就需要改进。
- 更小、边界更清晰的单元也更容易处理——你可以更好地理解那些能够一次性掌握上下文的代码，而且当文件处于焦点状态时，你的编辑也更可靠。当一个文件变得很大时，这通常表明它执行了过多的操作。

**在现有代码库中工作：**

- 在提出修改建议之前，先研究现有结构，遵循现有模式和架构。
- 如果现有代码存在影响工作的问题（例如，文件过大、边界不清晰、职责混乱），则应在设计中纳入针对性的改进——这是优秀开发人员改进他们正在编写的代码的方式。
- 不要提出与当前目标无关的重构建议。始终专注于能够帮助实现当前目标的重构。

## 关键原则

- 一次只问一个问题 - 不要一下子问太多问题
- 选择题优先 - 比开放式问题更容易回答
- YAGNI 原则 - 毫不留情地从所有设计中移除不必要的功能
- 探索其他方案 - 在最终确定方案之前，务必提出 2-3 种方案
- 增量验证 - 分章节呈现设计，并逐一验证
- 测试优先级 - 单元测试 > 集成测试 > 端到端测试

## 危险信号

千万不要跳过头脑风暴环节，因为：

- “这只是一个简单的组件” → 组件需要架构上下文
- “我已经知道要构建什么了”→ 用户可能有不同的预期
- “时间紧迫”→糟糕的设计会耗费更多时间
- “显而易见”→ 假设会导致错误的实现
